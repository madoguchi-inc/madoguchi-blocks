# madoguchi-blocks 自動更新機構（GitHub リリース連携）

## 要件定義

- 独自プラグイン `madoguchi-blocks` を、各サイトの WordPress 管理画面から**ワンクリック更新／自動更新**できるようにする。
- 配布先は**自社の複数メディアサイト**。
- 更新元は **GitHub リリース**。専用の**リリース配布用リポジトリ** `madoguchi-inc/madoguchi-blocks` を新設する（開発は monorepo `fuyouhin.support` 側で継続）。
- 配布リポジトリは **public** 運用とし、トークン設定を不要にする（private 切り替えの余地は残す）。

## 仕様

### 更新チェック

- ライブラリ [plugin-update-checker](https://github.com/YahnisElsts/plugin-update-checker) v5.7 を `madoguchi-blocks/lib/` に**同梱（vendoring）**。
- `madoguchi-blocks.php` の `madoguchi_blocks_boot_update_checker()` で初期化。
  - `PucFactory::buildUpdateChecker('https://github.com/madoguchi-inc/madoguchi-blocks/', __FILE__, 'madoguchi-blocks')`
  - `getVcsApi()->enableReleaseAssets()` … リリースのタグ名を新バージョン、添付zipを更新パッケージとして使用。
  - `MADOGUCHI_BLOCKS_GH_TOKEN` 定数がある時だけ `setAuthentication()`（public では未定義＝無認証）。
- WordPress 標準の `pre_set_site_transient_update_plugins` 経由で更新通知を注入するため、**プラグイン画面に標準の更新通知**が出る。「自動更新を有効化」ON で wp-cron による完全自動更新。

### バージョニング

- プラグインヘッダー `Version:` を上げる（1.1.0 → 1.1.1）。`Update URI:` ヘッダーも追加。
- **リリースのタグ名 = `Version:` ヘッダー**。ずれると更新判定が壊れるため厳守。

### 配布パッケージ

- `madoguchi-blocks/package.sh` が実行時ファイルのみ（`madoguchi-blocks.php` / `inc/` / `blocks/` / `build/` / `assets/` / `lib/` / `README.md`）を `madoguchi-blocks/` 構造で固めた zip を生成。
  - `src/` `scss/` `*.sh` は除外。`build/` 未生成時はエラー停止。
  - `--release` で `gh release create <version> <zip> --repo madoguchi-inc/madoguchi-blocks` まで実行。
- 出力先 `dist-zip/` は `.gitignore` で除外。

### リリースフロー

1. `bash madoguchi-blocks/build.sh`（build/ 最新化）
2. `Version:` を上げてコミット
3. `bash madoguchi-blocks/package.sh --release`
4. 各サイトに更新通知（最大約12時間のラグ、または「更新」ページで即チェック）

## 変更ファイル

| ファイル | 内容 |
|---|---|
| `lib/plugin-update-checker/` | 新規: PUC v5.7 同梱 |
| `madoguchi-blocks.php` | PUC 初期化追加、Version 1.1.1・Update URI ヘッダー |
| `package.sh` | 新規: 配布zip生成／リリース |
| `.gitignore` | 新規: `dist-zip/` 除外 |
| `README.md` | 自動更新の仕組み・リリース手順を追記 |

## 前提となる外部作業（未実施 / 別途要対応）

- **専用リポジトリ `madoguchi-inc/madoguchi-blocks` の新設**（public、初期コミット必須）。
  - リリースにはタグが指す commit が要るため、空リポジトリでは `gh release create` が失敗する。README だけでも1コミット入れて初期化する。
- 既存導入サイトへの、`lib/` を含む配布zip（v1.1.1）の初回反映。

## 検証項目

- [ ] `php -l madoguchi-blocks.php` / PUC ブートストラップの構文エラーなし（確認済み）
- [ ] `package.sh` が `dist-zip/madoguchi-blocks-<version>.zip` を生成し、トップが `madoguchi-blocks/`、`src/`・`scss/` を含まない（確認済み）
- [ ] 専用リポジトリ作成後、`package.sh --release` でリリースが作成され zip が添付される
- [ ] テストサイトに v1.1.0 を入れた状態でリリース v1.1.1 を公開 → プラグイン画面に更新通知が出る
- [ ] 「更新」実行で zip がDL・展開され、プラグインフォルダが `madoguchi-blocks/` のまま差し替わる
- [ ] 「自動更新を有効化」ON で wp-cron により無操作更新される
